# Brainstorm: ZPA Status Mini-SIEM Complete Architecture

**Date**: 2026-04-01
**Status**: Complete

## Context

The project replaces a legacy Cisco AnyConnect daily VPN report with a ZPA-based
solution. Beyond the initial report generator, the full system needs: production
deployment via install script, a web dashboard for the client's security team,
automated report delivery to a file share, and historical data access.

## Exploration Summary

Key insights from the discussion:

1. **Primary deliverable is the Excel report on the share** — the web dashboard is
   a secondary/extra feature for the client's security ops team, not for daily use.

2. **Installer must be interactive** — with clear prompts and sensible defaults, so
   it works both in a guided call and autonomously (open-source scenario). It must
   also generate a documented config file for later manual edits.

3. **Installer is re-runnable** — can be launched again (or via a separate command)
   to modify configuration interactively (cron schedule, share destination, etc.)
   without reinstalling everything.

4. **Distribution model is a zip file** — the client may have limited network access
   on the target machine. The entire project ships as a self-contained zip with
   the installer inside.

5. **Minimal dependencies** — Flask is acceptable (battle-tested, small footprint),
   but avoid heavy packages. SMB upload via `smbclient` CLI (already on RHEL),
   not Python SMB libraries.

6. **No NIC-specific binding** — everything listens on all interfaces (0.0.0.0).
   Network segmentation is the responsibility of the firewall administrator.

7. **Report retention: 6 months default**, configurable. Raw logs: 30 days (logrotate).

8. **Dual output**: Excel (for share delivery and download) + JSON (for web dashboard
   consumption and search).

## Approaches Explored

### Approach 1: Lightweight Monolith (Selected)
**Summary**: Single Python project with Flask. Report generator and web dashboard
share the same venv/codebase. Cron generates Excel + JSON. Flask serves the
dashboard with basic auth. SMB upload via `smbclient` CLI.

**Pros**:
- Single venv, single deploy, single systemd service for dashboard
- Centralized configuration in one documented file
- Easy to distribute as zip
- `smbclient` is already on RHEL, no extra Python packages for SMB
- Flask adds ~200KB, battle-tested for auth and routing

**Cons**:
- Flask is one extra dependency
- Dashboard and report generator are coupled

### Approach 2: Separated Services
**Summary**: Report generator as standalone cron script, web dashboard via
nginx/Apache with static files, share upload as separate script.

**Pros**: Independent components, nginx is robust for static serving
**Cons**: Three things to configure/maintain, more complex installer

### Approach 3: Monolith with Static Dashboard
**Summary**: No Flask — dashboard is a single HTML+JS file reading JSON,
served by Python's http.server.

**Pros**: Zero extra dependencies
**Cons**: No robust auth, limited search, more custom code = more bugs

## Decision

**Approach 1: Lightweight Monolith** — best balance of simplicity and functionality.
Flask is the right trade-off: one small dependency eliminates the need for
hundreds of lines of custom HTTP/auth code.

### Architecture Summary

```
Distribution:  project.zip (self-contained)
                  │
                  ├── install.sh (interactive installer + reconfiguration tool)
                  ├── config/    (rsyslog, logrotate templates)
                  ├── src/       (Python application)
                  └── requirements.txt

Installed on RHEL:
  /opt/zpa-siem/
  ├── venv/                    (Python virtual environment)
  ├── config.ini               (generated by installer, documented)
  ├── report_generator.py      (cron job)
  ├── web_dashboard.py         (Flask app, systemd service)
  └── reports/
      ├── 2026-04-01.xlsx      (Excel for download/share)
      ├── 2026-04-01.json      (JSON for web dashboard)
      └── ...

System integration:
  /etc/rsyslog.d/10-zpa.conf       (syslog reception)
  /etc/logrotate.d/zpa              (log rotation)
  /etc/systemd/system/
  ├── zpa-report.service            (oneshot report generator)
  ├── zpa-report.timer              (configurable schedule)
  └── zpa-dashboard.service         (Flask web dashboard)
```

### Configuration File (config.ini)

Generated by the interactive installer. All fields documented inline.
Re-configurable by re-running the installer or editing directly.

```ini
[general]
timezone = Europe/Rome

[syslog]
port = 514
protocol = tcp
log_dir = /var/log/zpa

[reports]
output_dir = /opt/zpa-siem/reports
schedule = 06:00
retention_days = 180

[dashboard]
enabled = true
port = 8443
username = admin
password = (hashed)

[share]
enabled = false
method = smb
# smb_share = //server/share/path
# smb_username = user
# smb_domain = DOMAIN
# scp_target = user@host:/path
```

### Installer Features

- Interactive prompts with sensible defaults
- Re-runnable for reconfiguration (`install.sh --configure`)
- Validates inputs (timezone exists, ports are valid, share is reachable)
- Generates documented config.ini
- Console UI with colors and clear feedback
- Idempotent — safe to run multiple times
